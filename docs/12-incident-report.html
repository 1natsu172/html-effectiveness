INC-2025-0412

タスク同期で502エラーが増加

SEV-2 解決済み 継続時間 47分 検知 4月12日 · 14:07 担当 Devon Park
TL;DR

設定のロールアウトによりsync-worker層のデータベース接続プール上限が64から8に下がり、通常の午後の負荷で接続が枯渇しました。同期APIは47分間で全リクエストの約21%に502を返しました。設定を元に戻してワーカーフリートを再起動することで影響を緩和し、データの損失はありませんでした。

タイムライン

14:02
設定変更cfg-9a12が標準のロールアウトパイプラインを通じて本番へ反映されました。
14:06
影響が発生。 同期ワーカーがプールのチェックアウト待ちで滞留し始め、p95レイテンシが4秒を超え、ロードバランサが502を返し始めます。
14:07
アラート発報: sync_5xx_rate > 2% が60秒間継続。オンコール担当（Devon）が確認。
14:18
初期の仮説はAPIサービスの不正なデプロイでした。直近2件のアプリケーションデプロイをロールバックしましたが効果はありませんでした。
14:31
Miraが加わり、ワーカーダッシュボードでプール待ちが飽和していることに気づきます。調査の焦点がアプリコードではなくインフラ設定へ移ります。
14:44
緩和完了。 cfg-9a12を元に戻し、ワーカーフリートを再起動。5xx率は3分以内に0.2%を下回りました。
14:49
モニターが5分間グリーンを維持。インシデントを解決と宣言し、ステータスページを更新しました。

根本原因

PR #4888により接続プール上限をワーカー層ごとに設定できるようになりました。新しいsync-workerキーの既定値はグローバル値（64）を継承するはずでしたが、ローカルテスト中に8へハードコードされたままコミットされていました。設定linterは型のみを検証し値の大きさは検証しないため、この変更はCIを通過しました。

設定のロールアウトとコードのデプロイは別々のパイプラインを通るため、オンコール担当が最初に取った対応、つまり直近のアプリケーションデプロイのロールバックは効果がなく、診断に約13分を費やしてしまいました。

infra/config/workers.yaml
pool:
global_max_connections: 64
tiers:
- sync-worker: { max_connections: 64 }
+ sync-worker: { max_connections: 8 } # debug value, do not ship
webhook-worker: { max_connections: 32 }

影響

失敗したリクエスト（502） ~41,200
ピーク時エラー率 21.4%
影響を受けたユーザー ~2,300 workspaces
データ損失 なし（クライアントが再試行）
SLA違反 なし（月次バジェット内）

アクションアイテム

DP cfg-9a12 を元に戻し、プール上限を64に復元する 4月12日
MO 設定linterにmax_connectionsの範囲チェックを追加（32未満で警告） 4月18日
SR オンコールダッシュボードでデプロイと並べて「最近の設定ロールアウト」を表示する 4月25日
PA フリート全体へ反映する前に、設定変更を1つのワーカーAZで10分間カナリア展開する 5月2日
