acme/api でのレート制限の仕組み

TL;DR すべてのリクエストは rateLimit() ミドルウェアを通過し、呼び出し元を バケットキー に解決し、Redis からトークンバケットを取得して、トークンを1つ消費するか 429 を返します。制限はルートごとに config/limits.yaml で宣言します。エントリのないルートは default ティア（APIキーあたり100リクエスト/分）を継承します。

リクエストの経路をステップごとに

各ステップを開くと、何が実行されどこにあるかを確認できます。経路全体はおよそ40行で、すべてのリクエストにp50でおよそ0.4 msを追加します。

1 · 呼び出し元を識別する middleware/ratelimit.ts:21

ミドルウェアはまずリクエストを bucketKey に絞り込みます。Authorization ヘッダがあればAPIキーを、なければクライアントIPを使います（x-forwarded-for チェーン経由で、信頼するのは自前のLBのみ）。匿名のIPトラフィックには、はるかに低いデフォルトティアが割り当てられます。

2 · バケットを参照する lib/tokenBucket.ts:9

ルート名とバケットキーの組み合わせが、tokens と updatedAt を保持するRedisハッシュ（rl:{route}:{key}）に対応します。キーが存在しない場合は、必要になった時点で満杯の容量で作成されます。ウォームアップはありません。

3 · 補充して消費する lib/tokenBucket.ts:31

補充量は経過時間から計算され（rate × Δt、上限は burst）、その後トークンが1つ減算されます。この読み取り・変更・書き込みの一連の処理は単一のLuaスクリプトとして実行されるため、同時リクエストが二重に消費することはありません。

4 · 空なら拒否する middleware/ratelimit.ts:48

スクリプトが tokens < 0 を返すと、ミドルウェアは 429 Too Many Requests で処理を打ち切り、Retry-After にトークンが1つ補充されるまでの秒数を設定します。成功したレスポンスには常に X-RateLimit-Remaining が付きます。

自分のルートに制限を設定する

ミドルウェアには手を加えません。config/limits.yaml にルート名をキーとしたエントリを追加し、（任意で）ミドルウェアが見つけられるようルートにタグを付けます。

# config/limits.yaml
default:
  rate: 100/min
  burst: 120

search.query:
  rate: 20/min
  burst: 40
  key: api_key        # or: ip
// routes/search.ts
router.post(
  "/search",
  rateLimit("search.query"),
  handler,
);
HTTP/1.1 429 Too Many Requests
Retry-After: 17
X-RateLimit-Limit: 20
X-RateLimit-Remaining: 0

{ "error": "rate_limited", "retry_after": 17 }
★
デフォルトティアだけで十分なら、YAMLエントリは一切必要ありません。ハンドラを引数なしの rateLimit() でラップするだけです。ルート名はパスから推測されます。

知っておきたい落とし穴

  • 開発環境では制限がプロセスごとになります。 REDIS_URL が未設定のとき、Redisクライアントはインメモリのマップにフォールバックします。そのためローカルテストでは実際のクラスタの挙動が再現されません。
  • Burst ≠ rate。 burst はバケットの容量です。1分間アイドルだった呼び出し元は、rate が低くても burst 個のリクエストを一気に送れます。
  • ストリーミングレスポンスは1回だけカウントされます。 トークンはリクエスト開始時に消費されます。30秒のSSEストリームでもトークンの消費は1つだけです。

FAQ

内部トラフィックを除外するには？
呼び出し元から x-acme-internal: 1 を設定します。ミドルウェアはそれをmTLSのピア名と照合し、バケットを完全にスキップします。
誰が制限されているかはどこで確認できますか？
429 が発生するたびに、ルートとキータイプのタグが付いた ratelimit.rejected メトリクスが送出されます。API → Health の下にGrafanaパネルがあります。
特定のユーザーだけ制限を高くできますか？
はい。YAMLの overrides: の下にそのユーザーのAPIキーを追加します。オーバーライドはデプロイなしで再読み込みされます。
