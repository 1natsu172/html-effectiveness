TL;DR

通知の送信はリクエスト経路の中でインラインに実行されていました。負荷がかかるとミューテーションのレイテンシに200〜800 msを上乗せし、SMTPプールが枯渇するとメールを黙って取りこぼしていました。このPRは配信を既存のpg-bossキューに移し、APIは即座に応答し、失敗した送信はバックオフ付きでリトライされるようにします。

なぜ

前四半期にコメントへ@メンションを追加したとき、1つのメンションごとに最大3つの送信（アプリ内・メール・Slack）が、コメントを保存するのと同じトランザクションの中で走るようになりました。リリース当初は問題ありませんでした。しかし今では1件のタスク更新が40人のウォッチャーへ波及しうるため、もはや看過できません。

Before
  • 送信がミューテーションハンドラ内でインラインに実行される
  • SMTPのタイムアウトがコメントの500エラーになる
  • リトライなし。取りこぼしたメールは失われたまま
  • comments.createのp99: 1.4 s
After
  • ハンドラは受信者ごとに1件のジョブをキューに入れて即座に返す
  • ワーカーが指数バックオフで3回リトライする
  • リトライ枯渇後に確認するためのデッドレターテーブル
  • comments.createのp99: 180 ms（ステージング）

このPRの中核です。notify.deliverジョブを取り出し、ユーザーのチャネル設定を解決して適切なアダプタを呼び出すpg-bossのサブスクライバーです。リトライはチャネルごとに設定します。メールは3回試行し、SlackはそのAPIが我々の側で既に冪等なので1回だけにしています。

boss.work('notify.deliver', { batchSize: 20 }, async (jobs) => {
  for (const job of jobs) {
    const { userId, event, channel } = job.data;
    const prefs = await getPrefs(userId);
    if (!prefs[channel]) return;          // user muted this channel

    try {
      await adapters[channel].send(userId, event);
    } catch (err) {
      if (job.retryCount >= MAX_RETRY[channel]) {
        await deadLetter(job, err);         // don't throw — ack & park
        return;
      }
      throw err;                            // pg-boss reschedules
    }
  }
});
