acme/web · アーキテクチャノート

認証がコードベースをどう流れるか

Acme はクッキーベースのセッションを使います。ブラウザがベアラートークンを直接保持することはありません。認証が必要なリクエストはすべて /api/* に届き、単一の verifyToken() ミドルウェアを通って Session 行へと解決されます。下流のハンドラはその行を req.ctx から読み取ります。ミドルウェアはセッションストアとやり取りする唯一の場所で、そのセッションストアは sessions テーブルとやり取りする唯一の場所です。そのため、考慮すべき信頼境界はちょうど1つだけになります。

src/app/providers/AuthProvider.tsx :22-48

マウント時、この React プロバイダは credentials: 'include' を付けて GET /api/session を発行し、fw_sid クッキーを一緒に送ります。レスポンスは currentUser をコンテキストにハイドレートするか、null のままにします。ルーターはこれを「サインイン画面を表示する」と解釈します。

ソースを表示 
// src/app/providers/AuthProvider.tsx
export function AuthProvider({ children }: Props) {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    fetch('/api/session', { credentials: 'include' })
      .then(r => r.ok ? r.json() : null)
      .then(setUser);
  }, []);

  return <AuthCtx.Provider value={{ user }}>{children}</AuthCtx.Provider>;
}

src/server/routes/session.ts :9-27

ルート自体は薄く、ミドルウェアが付与した req.ctx.session をそのまま返すだけです。ミドルウェアが 401 で処理を打ち切った場合、このハンドラは実行されません。そのため認証ロジックがここで重複することはありません。

ソースを表示 
// src/server/routes/session.ts
router.get('/session', verifyToken, (req, res) => {
  const { session } = req.ctx;
  res.json({
    id:    session.userId,
    email: session.email,
    role:  session.role,
    exp:   session.expiresAt,
  });
});
3
src/middleware/auth.ts :14-31

ここが信頼境界です。verifyToken は署名済みの fw_sid クッキーを読み取り、SessionStore にその解決を依頼し、req.ctx.session を設定するか 401 を返します。アプリ内の保護されたルートはすべてこの関数の背後にマウントされているため、この関数の挙動を変えると認証全体が変わります。

ソースを表示 
// src/middleware/auth.ts
export async function verifyToken(req, res, next) {
  const raw = req.signedCookies['fw_sid'];
  if (!raw) return res.status(401).end();

  const session = await SessionStore.get(raw);
  if (!session || session.expiresAt < Date.now()) {
    return res.status(401).end();
  }

  req.ctx = { session };
  next();
}

src/lib/sessionStore.ts :8-52

SessionStore は小さなリードスルーキャッシュです。まずプロセス内の LRU を確認し、なければ Postgres にフォールバックします。書き込み（create、revoke）は常に DB へ直接向かい、キャッシュエントリを無効化するので、他のワーカーが古いセッションを返すことはありません。

ソースを表示 
// src/lib/sessionStore.ts
const cache = new LRU<string, Session>({ max: 5000, ttl: 60_000 });

export const SessionStore = {
  async get(id: string) {
    const hit = cache.get(id);
    if (hit) return hit;
    const row = await db.one(SELECT_SESSION, [id]);
    if (row) cache.set(id, row);
    return row ?? null;
  },
  /* create, revoke, touch ... */
};
5
db/migrations/004_sessions.sql :1-18

sessions テーブルはランダムな 32 バイトの id（クッキーの値）をキーにし、「すべての端末からサインアウト」のために user_id へのカバリングインデックスを持ちます。有効期限はここ（expires_at）でもミドルウェアでも二重に強制され、多層防御になっています。

ソースを表示 
-- db/migrations/004_sessions.sql
create table sessions (
  id          text primary key,
  user_id     uuid not null references users(id),
  created_at  timestamptz default now(),
  expires_at  timestamptz not null,
  ip          inet,
  user_agent  text
);
create index sessions_user_id_idx on sessions(user_id);

主要ファイル

  • src/middleware/auth.ts リクエスト認証の単一エントリポイント。
  • src/lib/sessionStore.ts LRU と Postgres によるセッション参照。DB を呼ぶ唯一の場所。
  • src/server/routes/session.ts 現在のセッションをクライアントに返します。
  • src/server/routes/login.ts SessionStore.create を通じて認証情報をクッキーと引き換えます。
  • src/app/providers/AuthProvider.tsx サーバーのセッションを反映するクライアント側のコンテキスト。
  • db/migrations/004_sessions.sql sessions テーブルとインデックスのスキーマ。

落とし穴

  • SessionStore の LRU はプロセスごとに存在します。セッションを取り消してもローカルキャッシュがクリアされるだけです。他のワーカーは TTL が切れるまで最大 60 秒そのセッションを返し続けることがあります。
  • verifyToken は expiresAt を Date.now() と比較しますが、カラムは timestamptz です。ドライバが Date を返すため比較は成立しますが、チェックを調整せずに生の文字列へリファクタリングしないでください。
