ブラインドスポット一覧
地雷 01

セッションは二重に書き込まれる。最初に見つかるRedisストアは正しい方ではない

Acmeは、Postgresを使ったセッションからRedisへのマイグレーション途中です（MIG-118、3月から停滞中）。現在すべてのセッション書き込みはSessionBridge.write()を経由し、そこから両方、つまりRedisSessionStoreと旧来のpg_sessionsテーブルへ分配されます。マイグレーションは完了しないままなので、読み取りは今もPostgresから行われます。

なぜ痛いか「session」でgrepすると最初にRedisSessionStoreが出てきます。これが正式な実装に見えますし、最も新しいコードで、ドキュメントも一番充実しています。ここに直接書き込むと、開発環境ではログインが動いているように見え（開発ではRedisの読み取りが有効なため）、本番では読み取りが今もPostgresを見るため静かに失敗します。
新しいプロバイダのセッションを作成するときは、すべてSessionBridge経由にしてください。RedisSessionStoreやpg_sessionsに直接書き込まないでください。セッションのコードを書き始める前に、MIG-118の読み取り/書き込みの分離について説明してください。
地雷 02

SAMLプロバイダは、いかにも手本に見えて、実は認証ミドルウェアを迂回している

SAML以外のすべてのプロバイダはauthPipelineの配下にマウントされ、そこでレート制限、監査ログ、デバイス信頼チェックが実行されます。saml/handler.tsはACSエンドポイントをルーターに直接マウントしています。SP起点のPOSTコールバックが2024年にCSRF層を壊してしまい、誰も後で戻さなかったからです。14か月前の// TODO(priya): fold back into pipelineが今も残っています。

なぜ痛いかSAMLは最も最近追加されたプロバイダなので、コピー元として選びたくなる自然なファイルです。これをコピーすると、新しいプロバイダはレート制限も監査証跡もない状態で出荷されます。しかもどのテストも失敗しません。契約テストはミドルウェアの有無を検証していないからです。
構造の手本にはsaml/handler.tsではなくoauth/google.tsを使ってください。SAMLは意図的にauthPipelineを迂回しています。新しいプロバイダのルートがauthPipeline内にマウントされていることを確認し、そのマウント地点を見せてください。
履歴 03

これはすでに誰かが試みていて、今も有効な理由で差し戻された

PR #2841（「Okta OIDCプロバイダを追加」、2025年9月）はマージされましたが、9日後に差し戻されました。差し戻しコミットがその理由を説明しています。Acmeのコールバックリダイレクトのラウンドトリップを越えてもワークスペースに紐づく招待リンクが生き残るよう、コールバックURLはテナントのサフィックスを持たなければなりません（/auth/callback/:provider/:workspaceSlug）。Oktaの厳格なリダイレクトURIの一致判定はこの動的なサフィックスを拒否し、招待されたユーザーは間違ったワークスペースに着地しました。全員が合意した修正案は、AUTH_STATE_SECRETで署名したstateパラメータにワークスペースのクレームを載せる方式でした。差し戻しのスレッドで設計されましたが、実装はされませんでした。

なぜ痛いか現在のコードにはこれを示唆するものが何もありません。あなたは#2841のアプローチを再実装し、ローカルのテスト（単一ワークスペースの開発環境）をすべて通過し、招待されたユーザーが初めてサインインした瞬間に本番でこのバグを再発見することになります。
まずPR #2841の差し戻しを読んでください。テナントのサフィックス付きコールバックURLではなく、そこで提案されている署名付きstateパラメータのワークスペースクレームを実装し、ワークスペースに紐づく招待フローのテストを追加してください。
欠けている概念 04

アイデンティティとアカウントは別物である。しかもプロバイダインターフェースはそれを隠している

Acmeはusersとidentitiesを分けています。1つのアカウントに複数のアイデンティティ（メール+パスワード、Google、SAML）を連携でき、それらはidentitiesテーブルで結び付けられます。プロバイダインターフェースはプロフィールを返すだけで、連携の判断はidentity/linker.tsで検証済みメールをキーに行われます。検証済みメールを保証できないプロバイダ（多くのSSOのIdPは保証できません）は、requiresManualLink: trueを設定しなければなりません。そうしないと、企業の共通エイリアスを共有する別々の2人を、linkerが平然と統合してしまいます。

なぜ痛いかインターフェースを素直に実装すると、すべて動きます。ある顧客のIdPが未検証または再割り当てされたメールを送ってくるまでは。そのとき、あなたのプロバイダは既存のアカウントに赤の他人のアイデンティティを静かに紐づけてしまいます。これはログインのバグではなく、アカウント乗っ取りのバグです。
実装する前にidentity/linker.tsを順を追って説明してください。このIdPが検証済みメールを保証するかどうかを明示的に判断し、保証しないならrequiresManualLinkを設定して手動連携の確認フローを作ってください。
地雷 05

トークンのリフレッシュはフラグの背後にあり、開発ではオン、本番ではオフになっている

リフレッシュトークンのローテーションはflags.yamlのauth.refresh_rotationで制御されています。開発とステージングでは既定でtrue、本番ではfalseです（5月のクロックスキューのインシデント後に一時停止されたロールアウトのためです）。フラグがオフの本番では、セッションはちょうどACCESS_TOKEN_TTL（12時間）だけ生存し、その後は完全に失効します。リフレッシュの経路は存在しません。

なぜ痛いかあなたが書くリフレッシュ処理はどれも開発環境の挙動でテストされ、本番では死んだコードになります。さらに悪いことに、プロバイダがリフレッシュを前提にしていると、本番ユーザーは12時間ごとにセッション中でログアウトされ、バグ報告には「SSOがランダムにログアウトさせてくる」と書かれ、あなたのコードが疑われます。
auth.refresh_rotation=false（本番の実態）を前提に実装してください。プロバイダを12時間で完全失効する挙動で動くようにし、ローテーションのロールアウトが再開した場合に何が変わるかは別途書き添えてください。
慣習 06

プロバイダの登録には3つの手順があり、そのうち2つしか見つけられない

プロバイダはコンパイルが通っても稼働状態にはなりません。（1） providers/registry.tsに登録し、（2） マイグレーションでauth_providersテーブルに挿入し（管理UIはレジストリではなくDBを読むためです）、（3） __tests__/provider-contract.spec.tsのフィクスチャ一覧に追加する必要があります。このテストは、登録済みの全プロバイダに対して共通の契約スイート（state検証、リプレイ防止、ログアウトのファンアウト）を実行します。レジストリとフィクスチャ一覧が食い違うとCIが失敗します。

なぜ痛いかマイグレーションを飛ばすと、プロバイダはコンパイルが通り、テストも通り、それでいてどのワークスペースのSSO設定にも一切現れません。すでに2人のエンジニアがそれぞれ半日を無駄にしています。どのドキュメントにも書かれていないのです。
登録の3手順をすべて実施してください。registry.ts、auth_providersのマイグレーション、provider-contractのフィクスチャ一覧です。完了とする前に、新しいプロバイダで契約スイートが通ることを見せてください。
地雷 07

ログアウトは関数呼び出しではなくイベントであり、レビュープレーヤーがそれを購読している

セッションの取り消しは、内部のイベントバスにauth.session.revokedを発行します。リアルタイムのレビュープレーヤーは、それ自身が長寿命の再生権限を保持していて、そのイベントを受け取ったときにだけ権限を解放します。logout.tsは既存プロバイダについてこれを処理しますが、SSOのシングルログアウト（SLO）のコールバックは別の経路で到着するため、自らこのイベントを発行しなければなりません。

なぜ痛いかイベントを取りこぼすと、ログアウトは「動く」ように見えます。セッションの行は消えます。しかしレビュータブを開いたままの人は、権限のTTLが切れる（最大24時間）まで非公開の動画に完全な再生アクセスを持ち続けます。レビューのワークフローをスタジオに販売している会社にとって、これはバグではなくセキュリティインシデントです。
SLOに対応してください。新しいプロバイダのあらゆる取り消し経路はauth.session.revokedを発行しなければなりません。IdP起点のログアウトでレビュープレーヤーの権限が解放されることを検証するテストを追加してください。

ここまでの内容をすべて折り込んだ、本来あなたが私に渡すべきプロンプト

Acmeに新しいSSO認証プロバイダを追加してください。ブラインドスポット確認から得られた制約は次のとおりです。

1. セッション: すべてのセッション書き込みをSessionBridge経由にする
   （MIG-118は未完了で、本番は今もpg_sessionsから読む）。
   RedisSessionStoreに直接触れないこと。
2. 手本: 構造はoauth/google.tsを基にし、saml/handler.tsは使わない。
   SAMLはauthPipelineを迂回している。新しいルートはすべてauthPipeline内に
   マウントし、そのマウント地点を見せること。
3. 履歴: まずPR #2841の差し戻しを読む。テナントのサフィックス付き
   コールバックURLではなく、署名付きstateパラメータのワークスペース
   クレーム（AUTH_STATE_SECRET）を実装し、ワークスペース招待フローを
   テストする。
4. アイデンティティ連携: コーディング前にidentity/linker.tsを順を追って
   説明する。このIdPが検証済みメールを保証しないなら、
   requiresManualLinkを設定して確認フローを作る。自動連携はしない。
5. フラグ: auth.refresh_rotation=false（本番）を前提に実装する。
   セッションは12時間で完全失効し、リフレッシュ経路はない。
   ローテーションが出荷された場合に何が変わるかを書き添える。
6. 登録: 3手順すべて。providers/registry.ts、auth_providersの
   マイグレーション、provider-contractのフィクスチャ一覧。
   契約スイートがグリーンになるのを見せる。
7. ログアウト: あらゆる取り消し経路がauth.session.revokedを発行すること。
   IdP起点のログアウトでレビュープレーヤーの再生権限が解放されることを
   検証するテストを追加する。

stateクレーム → 登録 → SLO + イベント → 契約テスト。説明の後、コードを
書く前にいったん止まって計画を見せること。
