逸脱 2 / 415:22

キューは Date オブジェクトを文字列にして往復させる

計画の想定
ExportJobPayload.requestedAt: Date をワーカーまで引き渡し、レビューの updated_at と比較して古いエクスポートを検出する。
コードで判明したこと
BullMQ はペイロードを JSON にシリアライズして Redis に入れるため、ワーカーが受け取るのは Date ではなく ISO 文字列です。古さの比較は暗黙的に型変換されてユニットテストでは通っていましたが、統合テストの実行で再配信されたジョブでは失敗しました。
保守的な選択
ペイロードをワイヤーフォーマット（requestedAt: string、ISO-8601）として型付けし、exportConsumer.ts のワーカー境界で一度だけパースします。カスタムのシリアライザは使いません。
要再検討
他の media-tasks のペイロードも同じ潜在バグを抱えている可能性が高いです。ジョブペイロード用に共通の zod コーデックを用意すれば、この種の問題をまとめて解消できます。

ゲストのレビュアーはレビューを閲覧できるが、アセットはダウンロードできない

計画の想定
エクスポートの権限＝レビュー単位の ACL。レビューを開ける人は誰でもエクスポートできる。
コードで判明したこと
ワークスペース単位の guest_reviewer ロールは、can_download_assets = false の状態でレビューへのアクセスを許可します。顧客はこれを使い、メディアを外部に持ち出させることなく社外の関係者にコメントしてもらっています。レビューの ACL だけをチェックしていると、ゲストがエクスポート経由でソース動画を持ち出せてしまいます。
保守的な選択
エクスポートは guest_reviewer に対し全フォーマットで 403 を返します。これは assetAccess.ts にある既存のアセットダウンロードのチェックに合わせたものです。妥当な範囲で最も厳しい解釈です。
要再検討
ゲストでも CSV（注釈のみ、メディアなし）は取得できてよい、という考え方もあります。これは私ではなくプロダクト側が決めることなので、下に記録しておきました。
