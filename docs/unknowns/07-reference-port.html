1 このクレートが実際に行うこと
  • トークンバケット方式の受付（bucket.rs）: 送信されるリクエストは、rate_per_sec = 8 の速度で burst_capacity = 40 まで補充されるバケットから1トークンを引き出します。バケットは満杯の状態で始まるため、起動直後のクライアントは即座に40リクエストをバースト送信できます。
  • 補充は遅延評価で、整数に切り捨てられます: トークンはリクエストが到着したときだけ、経過ナノ秒 × レート ÷ 10⁹ から生成されます。端数のトークンは保存されません。代わりに last_refill は少なくとも1つの完全なトークンが生成されたときにしか進まないため、1トークンに満たない経過時間は次に持ち越されます。
  • 非相関ジッター付きバックオフ（backoff.rs）: リトライ可能な失敗が起きると、次の遅延は [base, min(prev × 3, cap)] から一様に選ばれます（base = 250ms、cap = 30s）。各回の抽選が次回の種になります。遅延は足並みをそろえて倍増するのではなくばらつきながら変動するため、障害後にクライアントの同期がずれます。
  • リトライ予算（budget.rs）: 2つ目の独立したバケットです。初回で成功したリクエストごとに1を預け入れ、リトライの試行ごとに10を引き出します。残高が引き出しをまかなえないと、リトライは完全に拒否されます（初回リクエストは引き続き送信されます）。これにより、障害が続く間もリトライの増幅を約10%に抑えます。
  • 失敗の分類は呼び出し側の仕事です: このクレートはエラーを一切検査しません。acquire()、next_delay()、try_withdraw() を公開し、何をリトライ可能とみなすかは呼び出し側が決めます。TS 版の移植でもこの境界を保ちます。Acme の src/api/errors.ts にある isRetryable() が唯一の信頼できる情報源であり続けます。

1時計が逆行する場合。 saturating_duration_since は負の経過時間をゼロに丸めます。Date.now() は NTP 補正で逆行することがあります。移植版は単調増加する performance.now() を使い、さらに念のため Math.max(0, …) も残して挙動を一致させます。
2整数演算と浮動小数点演算の違い。 Rust の u64 除算は切り捨てますが、JS の除算は切り捨てません。Math.floor が切り捨てを復元します。ナノ秒ではなくミリ秒を使っても安全です。rate = 8 のとき elapsedMs * 8 は 2⁵³ をはるかに下回るため、精度は失われません。
3要となるガード。 last_refill は完全なトークンが生成されたときにしか進みません。このガードを外すと（安易な「単純化」です）、呼び出しのたびに1トークン未満の進捗が黙って捨てられます。低いレートでは、頻繁にポーリングするとバケットがまったく補充されなくなります。そのまま正確に保ち、回帰テストも追加します。

4飽和乗算。 saturating_mul(3) は u64 のオーバーフローを防ぎます。JS ではこれはオーバーフローしません。capMs = 30_000 が積を 2⁵³ よりずっと手前で抑えるためです。そのためこのガードは意図的に外しています（§3 の「削除」列を参照）。
5閉区間か半開区間か。 Rust の lo..=hi は両端を含みます。単純な lo + random() * (hi - lo) は決して hi を返しません。Math.floor の中の + 1 が閉区間性を復元します。たった1文字でバグを招きやすい箇所なので、あなたが却下も承認もできるよう明示しています。
6状態を持つ種。 各回の抽選が次の反復の prev になります。これが、単なるジッター付き指数バックオフではなく非相関にしている仕組みです。reset() は成功時に prevDelayMs = baseMs へ戻し、クレートの Backoff::reset と一致します。this.random は決定的なテストのために注入可能です（クレートはテストで種を固定した SmallRng を使います）。

3 挙動: そのまま保持 / 意図的に変更 / 削除
そのまま保持
  • 補充の切り捨て + ガード — 完全なトークンのみ。lastRefill は生成時にしか進まない
  • ジッターの式 — [base, min(prev×3, cap)] の一様分布、両端を含む
  • 予算の収支 — 初回成功ごとに +1、リトライごとに −10、上限 1000
  • バケットは満杯で始まる — 起動直後の40リクエストのバーストは意図的（クレートのテスト burst_at_t0 と一致）
  • 呼び出し側がエラーを分類 — リミッターは失敗を一切検査しない
意図的に変更
  • Instant → performance.now() — どちらも単調増加。レート 8/s では ms 解像度で十分
  • u64 のナノ秒 → number の ms — すべての積は証明可能に < 2⁵³。Math.floor が整数除算を再現
  • Mutex<Budget> → 単なるフィールド — 原子性は同期のみという約束事 + テストで担保
  • SmallRng → 注入された random() — 既定は Math.random、テストでは種を固定したスタブ
削除（不要）
  • saturating_mul のオーバーフローガード — ms 範囲で cap を適用すれば到達しない
  • Send + Sync の実装、Arc のクローン — 共有すべきスレッドが存在しない
  • tokio/async-std のフィーチャーフラグ — 移植版は構造上ランタイム非依存
  • Prometheus のカウンター — Acme は代わりに telemetry.track() で送出。フック地点は残す
